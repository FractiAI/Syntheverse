# Documentation Standards

## Documentation Principles
- Write clear, concise, and accessible documentation
- Use understated language ("show not tell")
- Remove unnecessary adjectives
- Focus on practical information
- Keep documentation up-to-date

## README.md Structure

### Standard Sections
1. **Purpose** - What this component does
2. **Status** - Current operational status
3. **Architecture** - How it's structured
4. **Components** - Key modules/files
5. **Usage** - How to use it
6. **Integration** - How it connects to other parts
7. **Configuration** - Setup and configuration
8. **Testing** - How to test
9. **Next Steps** - Future improvements

## AGENTS.md Structure

### Standard Sections
1. **Purpose** - Component responsibility
2. **Key Modules** - Main files and their roles
3. **Integration Points** - How it connects to other components
4. **Development Guidelines** - Specific rules for this area
5. **Common Patterns** - Typical code patterns
6. **Dependencies** - What this component depends on

## Code Documentation

### Docstrings
- Use Google-style docstrings
- Document all public APIs
- Include parameter descriptions
- Include return value descriptions
- Document exceptions

### Comments
- Explain "why" not "what"
- Clarify complex logic
- Reference related code
- Keep comments up-to-date

## API Documentation
- Document all endpoints
- Include request/response examples
- Document error responses
- Include authentication requirements
- Use OpenAPI/Swagger when possible

## Architecture Documentation
- Document system architecture
- Include data flow diagrams
- Explain design decisions
- Document integration patterns
- Keep diagrams up-to-date

## User Guides
- Write step-by-step instructions
- Include screenshots when helpful
- Provide troubleshooting sections
- Keep guides current with UI

## Contributing Documentation
- Document contribution process
- Explain code review process
- Provide setup instructions
- Include coding standards
- Document release process
